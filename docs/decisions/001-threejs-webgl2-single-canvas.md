# ADR-001：使用 Three.js/WebGL2 单上屏 Canvas

## 状态

已接受；目标真机兼容性仍待最终构建验收。

## 日期

2026-07-14

## 背景

v2 使用 Canvas 2D、手写 2.5D 投影、烘焙光影贴图和精灵帧表现跳跃。这条路径可以跨 Web、微信与抖音共用业务逻辑，但无法稳定提供真实三维形变、统一光照、动态阴影、空翻和边缘倾倒。用户要求保留玩法内核，同时将其余表现按 `web-jump` 的立体动作方向重构。

项目又有两个平台约束：

- 微信与抖音小游戏不能依赖浏览器 DOM 叠加 HUD。
- 三端必须共用同一套场景、HUD 和输入命中语义，否则真机与 Web 验收会越来越分裂。

当前依赖树使用现代 Three.js 发布线（锁文件在本决策时解析为 `three@0.185.1`）。该路径以 WebGL2 上下文为运行前提，因此真机能力是发布门禁，不能只用桌面浏览器推断。

## 决策

1. 使用 Three.js `WebGLRenderer` 替换原 Canvas 2D 世界渲染。
2. 只使用 `webgl2` 上下文；平台层提供能力创建和校验，失败时返回包含平台标识的明确错误。
3. Web、微信和抖音都只创建一个上屏 Canvas。先渲染 World Scene，清理深度缓冲后再渲染 HUD Scene。
4. 允许用离屏 Canvas 生成字形、纹理或测量资源；离屏 Canvas 不参与上屏，不改变“单上屏 Canvas”决策。
5. 使用正交相机、环境光、单个受控方向光与有界阴影区域。
6. 保留 Vite + esbuild 的三端构建管线，不引入参考项目的 Webpack/Babel 配置。
7. 上下文丢失时停止提交新渲染；恢复后重建 GPU 资源并从 Core 快照重同步。

## 考虑过的替代方案

### 继续增强 Canvas 2D

优点是兼容性成熟且迁移成本低。缺点是需要继续手写投影、遮挡、形变和假阴影，无法解决本次重构最核心的动作质感问题，因此拒绝。

### World Canvas + DOM HUD

优点是 Web 文字与控件实现简单。缺点是小游戏没有等价 DOM，会产生两套布局、命中和视觉实现，因此拒绝。

### 两个上屏 Canvas

世界与 HUD 分开管理直观，但小游戏的上屏画布限制、层叠和输入转换会增加平台风险，因此拒绝。

### 回退到旧 Three.js/WebGL1

可以增加老设备覆盖，但会把已淘汰 API、安全修复和两套渲染分支带入长期维护。在没有真机失败证据前不预防性选择该路径。如果任一发行目标真机无法稳定使用 WebGL2，需新建 ADR 评估版本固定、平台分包或降级渲染，而不是在此 ADR 中预先承诺。

## 后果

- 可以使用真实 Mesh、统一材质和光照、动态阴影和分层角色 Rig。
- HUD 与世界在三端使用同一渲染与安全区语义。
- 阴影、多 Scene、资源 dispose 和上下文恢复增加了 GPU 生命周期复杂度。
- WebGL2 不再是可选增强，而是发布前必须在微信和抖音 iOS/Android 真机上证明的能力。
- 开发者工具、模拟器或本机浏览器的通过不能用于勾选真机项。

## 验收门禁

- Web 桌面与手机浏览器可创建 WebGL2，首帧无编译或链接错误。
- 微信与抖音开发者工具能导入最终构建，但该结果仅是真机前置检查。
- 微信和抖音的 iOS 与 Android 真机均稳定出首帧，无黑屏或持续上下文丢失。
- 阴影、透明资源、HUD、安全区、前后台和重建路径完成有证据验收。
