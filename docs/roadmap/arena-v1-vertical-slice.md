# Arena V1 完整分阶段实现路线

## 文档状态

已接受，2026-07-16。每个阶段必须先通过门禁再扩大范围；未通过时修复当前阶段，不把风险带入内容生产。

## 已锁定的首版范围

- 固定 1v1：一名玩家与一名隐藏本地机器人。
- 生产界面不展示机器人身份和难度，不提供难度选择。
- 简单、普通、困难由 match seed 按 `1:1:1` 等概率随机抽取。
- 三条命、最长 150 秒、无传统生命值，以装备、地图和击飞完成淘汰。
- 左侧移动、右侧一个上下文动作键；小地形允许自动通过。
- 首批 2 名角色、3 件装备、1 张完整地图。
- Web、微信、抖音三端；不包含后端、真人 PvP、排行榜或云存档。

## 阶段 0：文档、概念图与决策基线

交付：

- 产品愿景、游戏规则、角色索引、架构提案和 GitHub 调研。
- Q 版跑酷学徒与拟人发条方块机器人的概念图、模型拆解和动作拆解。
- `InputFrame`、只读快照、玩法事件、装备、地图、隐藏难度的概念合同。
- 当前数值跳台 v3 与 Arena V1 目标之间的明确边界。
- ADR-004：隐藏本地对手的 1v1 决策。

门禁：

- 文档不把 Arena V1 描述成已经实现。
- 难度随机、不可选择、生产 UI 不可见在所有目标文档中一致。
- 概念图仍保存在仓库，Markdown 本地链接和 Git diff 格式检查通过。

完成定义：已关闭。

## 阶段 1：竞技场最小骨架与物理 POC

状态：代码与本机门禁已完成。已选择轻量街机物理并记录 [ADR-005](../decisions/005-arena-lightweight-physics.md)；Rapier 候选已清理，Box3D 因 alpha 与分发成熟度未进入可执行 POC。目标真机加载证据待实机验收。

交付：

- 新建最小 `src/arena` 领域，不改写当前 v3 的 `src/core` 规则。
- 用相同测试场景比较项目内轻量街机物理、Rapier WASM、Box3D 3D WASM。
- 验证角色移动、地面判定、碰撞、显式冲量、平台边缘和淘汰面。
- 记录包体、初始化、固定步长稳定性、三端加载和调参复杂度。
- 新建物理路线 ADR；只为胜出方案增加正式依赖。

门禁：

- Web、微信、抖音开发构建均能加载 POC；至少完成目标真机的 WebGL/WASM 能力检查。
- 连续冲量、边缘接触和暂停恢复不产生 `NaN`、穿透或重复淘汰。
- 击飞可调且具有街机感，不被第三方刚体对象绑死为公共玩法 API。
- 性能测量包含逻辑、渲染、包体和启动成本，不只看桌面浏览器帧率。

## 阶段 2：无渲染 1v1 MatchCore

状态：已完成。38 项 Arena/架构测试通过，1,000 局压测全部结束，严格回放抽样一致；数据见 [MatchCore 压测结果](../research/arena-matchcore-stress-results.md)。

交付：

- 固定步长比赛状态机：准备、进行、重生、突然死亡、结算。
- 两名参赛者的移动、制动、朝向、碰撞、击飞、淘汰和重生。
- 三条命、150 秒计时和超时结算。
- match seed、具名 RNG 流、输入录制、回放和状态 hash。
- 只读 `MatchSnapshot` 与带稳定 ID 的权威玩法事件。

门禁：

- 无 Three.js 环境可以完成整局。
- 同 seed、同配置、同输入得到一致的淘汰、事件序列和最终 hash。
- 长时间、极端输入、暂停恢复和边界位置均有自动化测试。
- Renderer、动画和 HUD 无法修改命中、击飞、淘汰或胜负。

## 阶段 3：隐藏机器人与本地快速匹配

状态：代码与本机门禁已完成。148 项项目测试通过；10,000 个连续 seed 的三档比例均在目标范围内；900 局机器人压力验证完成且能力指标按简单、普通、困难递增。详细数据见 [机器人压测结果](../research/arena-bot-stress-results.md)，架构选择见 [ADR-006](../decisions/006-arena-local-tick-utility-bot.md)。生产匹配外壳和 Arena 可视入口仍属于阶段 6、8，不在本阶段伪造 UI 完成度。

交付：

- `QuickMatchService`：生成 seed，抽取虚构对手形象和隐藏难度，创建本地 `MatchSession`。
- 简单、普通、困难三套参数；追击、回中心、避险、威胁规避、重新站位和动作时机决策。争夺资源在阶段 4 的权威装备状态出现后接入。
- 人类化扰动：反应延迟、规划间隔、方向误差、短暂停顿、目标切换与风险偏好。
- 玩家和机器人统一为 `InputFrame`；机器人只能读取受限 `BotObservation`。
- 调试构建可查看或覆盖难度，生产构建不包含玩家可见入口。

门禁：

- 机器人不调用特权玩法接口，不读取未来掉落、机关结果或隐藏命中信息。
- 同一 seed 的对手形象、难度和机器人行为可以复现。
- 至少 10,000 个连续 seed 的三档分布各自落在 `33.3% ± 2%` 内。
- 对固定基准策略的批量结果能解释简单、普通、困难的行为差异；困难仍保留反应延迟和误差。
- 对手形象与难度无固定映射，切换难度不改变地图或装备随机序列。
- 生产 UI、无障碍文本和本地化文案不出现机器人或难度泄漏，也不包含“真人在线”、虚假在线人数或聊天。

本阶段实测只冻结“输入合法、行为可解释、难度能力递增”的骨架，不冻结最终玩家胜率。当前基础推击对固定基准存在站位与先手偏差，最终胜率必须等阶段 4、5 的装备和地图闭环形成后，在阶段 9 重新测量。

说明：纯客户端 H5 无法阻止高级用户反编译代码，因此这里保证的是生产体验和界面不主动披露，不把“不可逆向发现”作为虚假的安全承诺。

## 阶段 4：三件装备形成战斗闭环

已接受的实现默认值：动作冷却期间忽略动作输入，不回退为基础推击；死亡掉落优先放在最后安全位置，若无合法位置则记录诊断并回退到原始刷新点；盾牌只保护正面，正面命中取消锁链拉回并大幅降低击退，背面仍按正常规则结算，不提供无敌。

状态：Rule/Core/Bot 本机门禁已完成。基础推击与三件装备已统一到 ActionDefinition、ActionResolver、ActionExecutionSystem、Effect/Targeting Registry 和 EquipmentSystem；不存在第二套命中真相。Bot 只读延迟公开快照并输出同一 `InputFrame`。详细执行管线见 [Stage4 Rule/Core 执行管线](../architecture/arena-stage4-rule-pipeline.md)。

交付：

- 大锤：明显前摇、短距离、高击退。
- 锁链：中距离命中、拉回目标。
- 盾牌：短时防御、面对方向冲撞。
- 自动拾取、单装备槽、使用、冷却、打断、掉落和刷新。
- 机器人争夺、使用和规避装备。

门禁：

- 三件装备分别形成击退、拉扯、防守冲撞三种可辨识的站位决策。
- 装备命中只由 MatchCore 判定，动画不能补判、延长或取消命中。
- 玩家和机器人遵守相同拾取、前摇、冷却、硬直和防御规则。
- 任意装备组合都不会造成永久控制、无限位移或无法结束的比赛。

## 阶段 5：首张地图形成资源与淘汰闭环

状态：本机无渲染骨架和门禁已完成。首图使用 3×3 深渊平台、交替风场、30/60/90/120 秒装备波、60/90/110 秒分阶段塌陷，中心安全岛永久保留。地图选择已收口到共享 Authority Content/MapRegistry，批次使用恰好一次提交，Bot 按真实 surface 并集识别动态边缘。架构见 [Stage 5 地图权威执行管线](../architecture/arena-stage5-map-pipeline.md)，决策见 [ADR-008](../decisions/008-arena-map-authority-timeline.md)，证据见 [地图压测结果](../research/arena-map-stress-results.md)。

交付：

- 一张深渊竞技场。
- 分阶段塌陷地面、一个风场或旋转机关、一个预告式装备刷新点。
- 对局后半段的缩圈或加速塌陷。
- 数据化 `MapDefinition`、确定性事件时间轴和安全重生点验证。

门禁：

- 每个危险机制都有可读预警，出生和重生点不位于即时危险区。
- 装备不会刷新在不可达、已塌陷或淘汰区域。
- 批量 seed 中没有无法结束的地图状态，地图击杀来源可以统计。
- 地图 RNG 与对手形象、机器人难度和装备 RNG 相互独立。

## 阶段 6：触控灰盒可玩版

技术执行顺序与模块边界见 [Stage 6 输入、移动与灰盒执行计划](../architecture/arena-stage6-input-movement-plan.md)，逐项证据见 [Stage 6 验收与证据矩阵](../quality/arena-stage6-verification-matrix.md)，架构决策草案见 [ADR-009](../decisions/009-arena-semantic-input-and-movement-authority.md)。该 ADR 在 Rule/Core 骨架通过前保持提议状态，A/B 胜者在盲测后冻结。

状态：S6.1 候选的 Character/Input/Replay 合同和无渲染兼容门禁已通过；尚未进入 Movement Rule/Core，更不表示触控、灰盒或盲测完成。

交付：

- 左侧二维移动与右侧一个上下文动作键。
- 稳定的动作优先级、当前行为提示、自动拾取和小地形自动通过。
- 玩法级走、跑、起跳、蹲跳、二段跳、下砸语义的控制原型。
- 正交斜俯视跟随相机、基础 HUD、匹配转场、结算和“再来一局”。
- 程序化代理角色、灰盒地图和装备占位表现。

门禁：

- 首次玩家在 10 秒内能移动、理解动作键并进入第一轮争夺。
- 单按钮下的攻击、装备、跳跃、二段跳、下砸和互动不存在无法解释的冲突。
- 通过盲测决定主动跳跃/二段跳与自动跨越的最终映射，并同步回游戏规则；这是进入正式动画生产前的必要决策。
- 从快速匹配到结算可以连续完成多局，没有比赛状态或资源泄漏。

## 阶段 7：角色、动画与反馈系统

技术执行顺序与资产/生命周期边界见 [Stage 7 角色、动画与反馈执行计划](../architecture/arena-stage7-presentation-plan.md)，架构决策草案见 [ADR-010](../decisions/010-arena-semantic-presentation-and-assets.md)。在 Stage 6 输入语义冻结前不进入批量正式动画生产。

交付：

- Q 版跑酷学徒与拟人发条方块机器人两名正式角色。
- 低多边形玩具主体、手稿风线条；衣服、翅膀、挂件和拖尾采用组合外观插槽。
- 六方向朝向，以及待机、走、跑、起跳、蹲跳、二段跳、下砸、落地、受击、击飞、淘汰、装备和胜负动作。
- 纯外观层、表现性动作层、玩法级动作层保持明确分离。
- 命中线、拖尾、击飞线、地图预警、镜头和音频反馈。

门禁：

- 关闭动画、特效和音频后，MatchCore 结果与回放 hash 不变。
- 两名角色在相同玩法模板下通过同一物理基准，外观挂件不改变碰撞体。
- 六方向轮廓、装备持有状态、前摇和危险预警在手机尺寸下可读。
- GLB 通过格式校验，动画、材质、纹理、实例数和内存满足资产预算。
- “减少动态效果”关闭强震动、过强拖尾和大幅镜头反馈，但不删除必要规则提示。

## 阶段 8：完整局外产品循环

产品状态机、双槽本地存档、迁移、幂等奖励和对称内容池见 [Stage 8 局外产品循环与本地进度执行计划](../architecture/arena-stage8-product-progression-plan.md)，决策草案见 [ADR-011](../decisions/011-arena-versioned-local-progression.md)。

交付：

- 角色选择或随机、快速匹配、短准备、对局、结算、奖励、解锁和再来一局。
- 虚构对手昵称、头像、外观和准备状态内容池；不与难度固定绑定。
- 本地进度、设置、教程提示和减少动态效果。
- 外观、角色、装备图鉴和地图解锁；不提供开局数值优势。

门禁：

- 生产流程没有难度选择、机器人标签或难度结算信息。
- 所有对手资料均为虚构内容，不冒用真实账号，不伪造真人社交数据。
- 新装备解锁只扩展玩家与机器人共享的对称掉落池。
- 首次启动、升级存档、损坏存档恢复、切后台和重开均有确定行为。

## 阶段 9：平衡、可靠性与性能收敛

可复现实验、黄金回放、输入模糊、长稳和表现降级见 [Stage 9 平衡、可靠性与性能收敛计划](../architecture/arena-stage9-convergence-plan.md)，决策草案见 [ADR-012](../decisions/012-arena-reproducible-convergence.md)。

交付：

- 批量 seed 模拟与指标面板：三档分布、胜率、平均时长、自杀率、装备争夺率、淘汰来源。
- 固定回放样本、回归用例、输入模糊测试和长时间 soak test。
- Web、微信、抖音的启动、内存、帧时间、对象数、恢复和触摸验收。
- 对局节奏、装备参数、地图事件和机器人参数的配置冻结候选。

门禁：

- 主要对局时长分布落在 2～3 分钟，异常超短和超时局有可解释上限。
- 三档难度对固定基准的表现有稳定顺序，但不存在作弊式瞬时反应。
- 连续多局、前后台切换和上下文恢复没有持续资源增长或状态串局。
- 三端目标设备达到项目设定的启动、帧时间、内存和包体预算。
- 所有阻断级与高优先级缺陷关闭，平衡变更都有批量证据或可重复实机观察。

## 阶段 10：发布候选与冻结

交付：

- 唯一 RC 构建、版本号、变更记录、已知问题和回滚构建。
- 三端最终包、平台配置、权限与隐私文案、素材与第三方许可清单。
- 首次用户、连续多局、异常恢复和发行前冒烟测试记录。
- 冻结的规则、配置、内容清单和对应 Git tag/commit。

门禁：

- RC 使用与提交记录一致的源代码和锁定依赖，可重复构建。
- Web、微信、抖音开发者工具与目标真机验收均留有证据。
- 没有把本地快速匹配写成真人在线，也没有未授权素材或真实账号冒用。
- 回滚路径经过演练；发布后新增范围进入下一版本，不直接修改已冻结规则。

## 当前不阻塞、但必须在对应阶段决策的事项

完整的决策时点、备选与推荐默认见 [Stage 5～9 决策门](stage5-9-decision-gates.md)。以下事项不应在没有对应证据时凭文档冻结：

1. 阶段 6：在单动作键约束下，主动起跳、蹲跳、二段跳、下砸与装备技能的最终输入映射。
2. 阶段 9：三端具体帧时间、内存、包体和启动预算，以目标机型实测后冻结。

## 延后项目

以下不属于 Arena V1：

- 真人 PvP、网络房间匹配、P2P 或服务器权威联机。
- 排行榜、赛季、账号社交和云端存档。
- 角色专属主动技能、复杂连招和超出首版清单的大量装备或地图。

如果未来进入真人对战，必须新建网络 ADR 和延迟、作弊、重连、账号、隐私与服务器运维验收计划，不能默认复用本地机器人架构。
