# ADR-004：采用分批治理的模块化单体

## 状态

Accepted

## 日期

2026-07-15

## 背景

当前项目已具备纯 Core、固定步长 Runtime、Three.js Renderer 和三端平台适配，但规则、难度、应用编排、场景、角色、反馈和存档尚未形成稳定的可替换契约。`runtime/game.js`、`renderer3d.js` 和平台适配器承担较多职责，项目也尚未启用 TypeScript、覆盖率和 CI 门禁。

项目需要长期支持至少 5 个玩法定义、5 个任务定义和 10 个角色 Manifest，同时保持单机、低耦合、高内聚和可独立测试。

## 决策

采用单仓库、单游戏、private npm workspaces 的模块化单体，并分五次交付：

1. 第 0 批：文档和事实基线。
2. 第一批 P0–P2：基线、契约、workspaces 和难度。
3. 第二批 P3–P5：Jump Engine、Gameplay/Task 和 Application。
4. 第三批 P6–P8：Renderer、场景/角色和 Feedback。
5. 第四批 P9–P10：存档/回放、全面 TypeScript 和工程门禁。

每批必须保持三端可构建，经过健壮性、竞态、兜底、边界、Web 生命周期和主流程审计，修复阻断问题，校准文档，中文提交并推送后才能进入下一批。第四批后再执行独立终验。

新模块直接使用 TypeScript；旧 JavaScript 渐进迁移。第四批结束时，人工维护的业务源码、测试和构建逻辑不得保留 JavaScript。

## 备选方案

### 一次性全量重写

- 优点：目标目录可以一次成型。
- 缺点：架构、类型、构建和玩法回归同时发生，难以定位差异，长期分支不可发布。
- 结论：拒绝。

### 只在现有 `src` 中增加子目录

- 优点：迁移成本低。
- 缺点：无法通过 package exports、独立 package.json 和依赖图强制边界，长期容易重新耦合。
- 结论：作为短期迁移状态允许，最终采用 workspaces。

### 拆成多个独立仓库

- 优点：物理隔离强。
- 缺点：本游戏的玩法、Renderer、内容和平台需要原子变更，多仓版本协调成本高。
- 结论：拒绝。

### 微服务或运行时远程插件

- 优点：独立部署和动态扩展。
- 缺点：项目是离线优先的单机游戏，不需要服务端并发；远程代码增加安全、审核和可复现风险。
- 结论：拒绝。

## 影响

- 治理期间会暂时存在新 workspace 包与旧 `src` 兼容层，但每批必须明确删除节点。
- 构建脚本需要逐步适配 TypeScript 和 workspace 源码。
- 测试从当前 87 项扩展为单元、契约、集成、回放、soak 和浏览器矩阵。
- 模块公共 API、配置和存档都需要版本。
- 文档和 ADR 成为每批提交门禁的一部分，不再是事后补充。
