# ADR-005：正式内容目录与任务驱动胜负

## 状态

Accepted

## 日期

2026-07-15

## 背景

治理阶段只证明注册表能容纳 5 个玩法、5 个任务和 10 个角色。发布内容仍只有一个玩法、一个任务和一个角色，并存在三个结构性缺口：

- 不同 Gameplay 可以创建不同 `GameState`，但内置注册表只有一个正式定义。
- Task 会被评估和发出事件，但默认胜负仍由 `GameState` 的“精确命中”条件提前决定。
- Character 只有颜色与缩放差异，容量测试使用 `fixture-character-*`，不是正式角色内容。

同时，Web、微信和抖音必须继续共用单上屏 Canvas，不能为 Web 单独增加 DOM 菜单。

## 决策

1. 在同一数值策略跳跃应用族内交付 5 个正式 Gameplay。每个玩法通过版本化定义变换规则，不复制主循环。
2. 交付 5 个正式 Task。每次成功落地后由选中 Task 返回 `active/completed/failed`，`GameState` 只负责应用该结果。
3. Gameplay 显式声明兼容 Task；内容菜单只循环兼容项，不用 ID `if/else` 修补组合。
4. 交付 10 个正式程序化 Character。`CharacterAppearance` 声明身体轮廓、附件、动作风格和副色，Three Rig 根据定义组装几何与动作。
5. 玩家通过 Three.js HUD Scene 内的内容菜单选择玩法、任务和角色；菜单、游戏 HUD 与世界继续渲染到同一 WebGL2 Canvas。
6. `SaveEnvelope` 升级为 v4，在游戏身份中保存角色定义。v1、v2、v3 存档迁移时使用 `jumbo-red@1`。
7. 本轮“解锁”表示全部正式内容可选择，不引入等级、付费、账号或进度门槛。

## 备选方案

### 为每个玩法复制一套 Runtime

- 优点：单个玩法实现自由。
- 缺点：输入、固定步长、物理、存档和生命周期会形成五套真相。
- 结论：拒绝。同一交互家族应共享 Application 与 Jump Engine。

### 保留 Task 仅用于徽章或事件

- 优点：无需改变落地结算。
- 缺点：五个任务不会改变完成条件，属于名称不同的假内容。
- 结论：拒绝。任务必须拥有完成与失败语义。

### 使用 Web DOM 菜单

- 优点：实现和排版简单。
- 缺点：微信/抖音需要第二套界面与命中逻辑，破坏单 Canvas 约束。
- 结论：拒绝。菜单进入共享 HUD Scene。

### 立即引入外部角色模型

- 优点：美术细节上限更高。
- 缺点：增加包体、加载失败、授权、骨骼动画和真机 GPU 风险。
- 结论：本轮拒绝。先用可审计、可销毁、零外链的程序化角色建立正式内容基线。

## 影响

- 新 Gameplay、Task、Character 必须提供玩家可见名称和描述。
- 新任务需要确定性单测，且不能绕过落地事务修改世界状态。
- 新玩法必须证明每个起始局至少能生成两个合法候选；正式发布前仍需做更大样本可解性验证。
- 角色切换是资源替换事务，失败时回退默认角色并释放旧 Rig。
- v4 存档仍是本地确定性动作日志；本轮没有新增个人信息或云服务。
