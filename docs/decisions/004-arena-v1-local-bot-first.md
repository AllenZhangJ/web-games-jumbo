# ADR-004：Arena V1 首版采用隐藏本地对手的 1v1

## 状态

已接受。

## 日期

2026-07-16

## 背景

Arena V1 的核心风险是装备、地图、站位和击飞是否能在极简操作下形成持续策略，而不是真人匹配系统是否可用。若第一阶段同时建设房间、匹配、网络预测、重连、作弊防护和账号服务，会延迟对核心玩法的验证，并扩大 Web、微信和抖音的兼容范围。

用户明确要求第一版本不做真人对战，而是营造 1v1 快速匹配的体验。对手由本地机器人控制；简单、普通、困难三档按局随机，玩家既不能选择，也不能在生产界面看到身份或难度。

## 决策

1. Arena V1 固定为一名玩家与一名本地机器人进行 1v1。
2. 产品入口使用“快速匹配”“寻找对手”等中性语言；生产界面不展示机器人身份，也不提供难度选择。
3. 简单、普通、困难由 match seed 按 `1:1:1` 等概率抽取。对手形象与难度使用独立随机流；难度只写入调试、自动化统计和回放元数据。
4. 不引入 Colyseus、Nakama、Socket.IO、WebRTC、P2P、匹配服务器、账号或数据库依赖。
5. 玩家和机器人都通过同一个 `InputFrame` 合同驱动 MatchCore。
6. 机器人难度来自反应延迟、决策频率、方向误差、风险偏好和目标权重，不来自隐藏信息或玩法特权。
7. MatchCore 保持固定步长、seed 和回放能力，为批量机器人测试及未来网络研究保留基础。
8. 本地奖励和解锁使用现有 Platform Contract 的存储能力。
9. 不使用“真人在线”等明确断言，不伪造在线人数、聊天、好友、位置或账号历史。对手昵称和头像来自虚构内容池，不冒用真实玩家身份。

## 考虑过的替代方案

### 首版直接做服务器权威真人 PvP

能够更早验证人与人互动，但会同时引入网络延迟、服务器部署、重连和作弊防护，难以判断手感问题来自玩法还是网络，因此拒绝用于 Arena V1。

### 使用 WebRTC P2P 锁步

可以减少服务器基础设施，但 NAT、TURN、断线恢复、小游戏宿主兼容和作弊边界仍需解决，而且不能替代机器人玩法验证，因此拒绝用于 Arena V1。

### 让机器人直接调用玩法 API

实现速度快，但机器人会绕过真实输入、冷却和动作约束，难度无法代表真实玩法，也无法公平调试，因此拒绝。

### 向玩家提供难度选择

便于定向练习，但会破坏首版“发起匹配、面对未知对手”的体验，也会把内部 AI 参数变成显式模式，因此拒绝用于生产流程。调试构建仍可保留受保护的难度覆盖入口。

### 声称对手是真人或伪造社交信号

短期可能增强匹配感，但会制造无法兑现的产品承诺，并给后续客服、平台审核和真实联机升级带来信任成本，因此拒绝。首版只隐藏实现身份，使用中性竞技文案。

## 后果

- 可以先验证装备、地图和击飞循环，减少首版系统数量。
- 可以用无渲染批量模拟量化机器人自杀率、胜率、装备争夺率和对局时长。
- 第一版无需运行后端，三端构建仍是纯客户端产物。
- 机器人需要清晰的受限观察接口、输入合同和独立 RNG 流。
- 产品需要一套不依赖真人声明的虚构对手昵称、头像和准备状态资源。
- 难度分布按 seed 可审计，但生产 UI 不显示该信息。
- H5 客户端代码可能被高级用户检查，因此只能保证正常产品体验不主动披露，不能承诺实现身份在逆向分析下永远不可发现。
- 未来若增加真人 PvP，必须新建 ADR 选择网络模型，并补充延迟、预测、作弊、重连和服务器运维设计；本 ADR 不预先承诺技术栈。

## 验收门禁

- 简单、普通、困难三档在批量对局中表现出可解释差异。
- 三档在大样本 seed 中接近 `1:1:1`，同一 seed 总能得到同一难度。
- 生产路径没有难度选择控件、机器人标签或难度标签。
- 对手形象与难度没有固定映射，地图与装备序列不受难度变化影响。
- 所有匹配文案均不宣称真人在线，也没有伪造的在线或社交数据。
- 任一难度都不读取未来随机事件或直接修改 MatchCore。
- 固定 seed 和输入可以重放机器人对局。
- 1 名玩家 + 1 名机器人在目标设备性能预算内稳定完成多局。
