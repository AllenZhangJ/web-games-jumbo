# ADR-005：Arena V1 采用轻量街机物理

## 状态

已接受。

## 日期

2026-07-16

## 背景

Arena V1 是固定斜俯视的 2.5D 跑酷竞技游戏。首版只有两名直立角色、静态平台、地图淘汰和显式击飞，要求 Web、微信和抖音保持高帧率、快速启动、可回放和易调参。物理后端不能把第三方对象暴露给 MatchCore。

阶段 1 使用统一 `PhysicsAdapter` 对项目内轻量 JavaScript、Rapier 3D deterministic compat 0.19.3 进行了相同场景比较。完整结果见 [`../research/arena-physics-poc-results.md`](../research/arena-physics-poc-results.md)。

## 决策

1. Arena V1 使用项目内轻量街机物理。
2. 角色使用直立胶囊语义，锁定倾倒；移动在 x/z 平面，y 用于重力、跳跃和击飞。
3. 物理固定为 60 tick/s，轻量后端隐藏在 `PhysicsAdapter` 后面。
4. 第一版只承诺数据化轴对齐静态平台、低台、角色互撞、显式冲量、地面吸附和淘汰面。
5. MatchCore、Renderer 和内容定义只能交换普通数值对象，不持有物理内部引用。
6. 逻辑物理的目标预算为每 tick 不超过 1ms；最终结论必须由目标真机复核。

## 考虑过的替代方案

### Rapier 3D deterministic compat

优点是成熟碰撞、CCD、快照和跨平台确定性构建。它通过了有限数、角色互撞、击飞、跌落和重置场景。

拒绝原因：本机 POC 中 gzip 为 844,292 B，轻量方案为 2,781 B；平均 step 约高 10.7 倍。统一动态胶囊控制不能稳定通过 0.3m 低台，需要额外角色控制器集成，增加接地、冲量和动态角色交互的调试面。

### Box3D

具备 C17、跨平台确定性、记录和回放等能力，但当前仍是 alpha，缺少成熟官方 JavaScript/WASM 分发。为 H5 首版自建绑定的成本和风险超过当前玩法需求，因此不进入可执行 POC。

### 完整自研通用刚体引擎

可以最大化控制权，但首版不需要旋转刚体、关节、复杂网格和通用求解器。开发通用引擎会扩大验证范围，因此拒绝；只实现 Arena 已确认规则需要的窄物理能力。

## 后果

- 包体、启动和每 tick 成本保持很低，适合为渲染、AI、特效和小游戏宿主开销预留预算。
- 击飞、制动和接地完全可按游戏手感调节，不追求现实刚体。
- 坡面、移动平台和旋转机关需要以后按实际地图需求扩展适配器并新增确定性测试。
- 轻量物理代码属于权威玩法，需要严格输入校验、边界测试、长稳压测和回放 hash。
- 若未来地图需要复杂网格或大量动态刚体，应新建 ADR 重新评估第三方物理，而不是在现有实现中无界扩张。

## 验收门禁

- 统一 POC 全部场景通过，100,000 tick 无非有限状态。
- Web、微信和抖音 POC 均能构建为无外部模块的 ES2020 IIFE。
- 两名角色的 MatchCore 批量压测保持每 tick 物理预算小于 1ms。
- 相同初态和输入得到相同最终状态 hash。
