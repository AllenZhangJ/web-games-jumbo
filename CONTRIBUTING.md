# 贡献指南

## 基本原则

- 当前仓库只服务“数域跃迁”，不要把其他游戏放进本仓库。
- 玩法结果只由 Gameplay、Task、Difficulty 和 Jump Engine 决定。
- Three.js、场景、角色、声音和震动不能反写玩法真相。
- 平台 API 只能存在于平台适配器和入口组合根。
- P0–P8 治理期间冻结当前玩法、碰撞和手感；行为变化需要单独批准。

## 开发流程

1. 从最新 `main` 创建对应批次或功能分支。
2. 先更新/新增测试，再完成最小可验证改动。
3. 运行相关测试和 `npm run check`。
4. 按 [批次提交前检查清单](docs/governance/batch-checklist.md)完成审计。
5. 更新当前事实文档、未完成项和不确定项。
6. 检查暂存文件、许可证和敏感信息。
7. 使用中文提交信息并推送。

## 提交信息

建议格式：

```text
治理：建立第 0 批项目文档基线
重构：提取确定性跳跃引擎
测试：补充生命周期竞态回归
修复：处理销毁后的异步启动回写
文档：校准第二批模块职责和未完成项
```

提交应表达一个完整意图，不混合无关重构、格式化和产品功能。

## 验证

当前最低门槛：

```bash
npm test
npm run build
```

治理过程中新增的 lint、typecheck、coverage、contract、integration、replay 和 soak 命令一旦进入主分支，就成为后续提交的累积门槛。

## 文档

- 从 [项目文档索引](docs/README.md)开始。
- 重大且昂贵的决定写 ADR，不覆盖或删除旧 ADR。
- 文档必须区分“当前、目标、待验证、未完成”。
- 代码与文档冲突时先以代码和测试为证据，然后在当前批次修正文档。
