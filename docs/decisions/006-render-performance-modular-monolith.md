# ADR-006：渲染性能治理采用模块化单体与资源预算

## 状态

Accepted

## 日期

2026-07-15

## 背景

正式内容扩展到 5 个玩法、5 个任务和 10 个角色后，内容菜单以状态组合为 key 创建 `1220×1220` CanvasTexture。原 TextureManager 只按 96 个条目限制缓存，不按字节限制；遍历 10 个角色即可保留 59,536,000 bytes 的菜单像素，桌面诊断曾记录总缓存约 81 MB。手机在松开按钮、角色由蓄力切换到起跳和空翻的同一时刻出现明显停顿。

项目仍是单机 Three.js 小游戏，渲染代码只有一个消费者。立即拆成多个发布包或引入 ECS 会扩大迁移面，却不能直接解决纹理上传、热路径分配和资源生命周期问题。

## 决策

1. 保持 private workspace `renderer-three`，在包内建立 `facade`、`frame`、`resources`、`hud`、`effects`、`scene`、`character`、`world`、`diagnostics` 边界。
2. Renderer 继续只消费 Snapshot/Event；FrameCoordinator 是每帧更新顺序的唯一所有者。
3. 所有 GPU/Canvas 资源必须有一个明确的 RenderResourceScope 所有者。
4. UI 纹理缓存按估算字节和引用状态治理，不再只按条目数治理；内容菜单使用固定动态纹理而不是组合缓存。
5. 特效通过注册表和 EffectRuntime 扩展，运行期复用对象，不在起跳热路径分配 Three 对象。
6. 提供用户可选的 `high`、`low` 两档。两档均以 60 FPS 为目标；低档通过 DPR、阴影和特效容量降载，不主动降到 30 FPS。两个档位的释放响应预算均为 33 ms，长任务上限为 50 ms。
7. 存档写入移到已提交动作后的下一渲染帧；Hide、PageHide 和 Destroy 强制刷新。异常强杀可能丢失最后一个尚未刷新的动作，该取舍已由项目负责人接受。
8. 依赖、热路径分配、字节预算、泄漏、首跳、100 局 soak 和内容全遍历守卫进入强制门禁。

## 备选方案

### 立即拆成多个 workspace

- 优点：物理目录边界更明显。
- 缺点：只有一个消费者，发布和构建成本增加；资源生命周期仍需统一内核。
- 结论：暂不采用。模块超过 800 行并持续增长、出现两个以上消费者或需要独立发布时再评估。

### 引入 ECS 或替换 Three.js

- 优点：大规模实体场景扩展能力更强。
- 缺点：当前瓶颈是 HUD CanvasTexture 和同步热路径，改引擎不能自动解决，且会危及已冻结玩法与碰撞。
- 结论：拒绝。

### 继续只按缓存条目数治理

- 优点：改动最小。
- 缺点：无法区分 `8×8` 与 `1220×1220` 纹理，96 个大纹理远超移动设备预算。
- 结论：拒绝。

## 后果

- 渲染内部 API 增加，但不改变 Gameplay、Jump Engine、碰撞或存档格式。
- 高低画质的视觉能力必须显式配置；低档可减少阴影、粒子和拖尾，但不能改变 HUD 可读性和游戏结果。
- 每个性能批次必须同时提供代码、测试、当前事实文档和可追溯验收证据。
- 微信/抖音构建继续保持，但对应真机性能不能由 Web 或桌面结果代替。

2026-07-15 本机高刷复测发现 30 FPS 目标会把可避免的运动不连续定义成正常行为，因此将低档目标修订为 60 FPS；本修订不改变任何玩法或碰撞结果。
